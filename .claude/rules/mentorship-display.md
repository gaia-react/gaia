# Mentorship Display

Mentorship raw event files contain identity-bearing data — git author, machine paths, hostnames preserved in `_local` namespace per `studio/decisions/telemetry-v1-design.md`. Claude must not display, summarize, tail, or otherwise surface their contents to the user. Only `profile.md` (the regenerated aggregate) is Claude-displayable. Raw events are accessible to the user via the `gaia mentorship` CLI surface, never via Claude.

## Scope

The contract covers:

- Files under `~/.claude/projects/<slug>/gaia/telemetry/mentorship/events-*.jsonl`
- Any file under `~/.claude/projects/<slug>/gaia/telemetry/mentorship/` regardless of name
- Lines piped from those files into Claude via Bash output

## What Claude does on request

When asked to read, show, summarize, tail, grep, or otherwise display the contents of those files, refuse and surface two pointers:

- `profile.md` — the Claude-displayable aggregate at `~/.claude/projects/<slug>/gaia/profile.md` (regenerated by `gaia telemetry compute-profile`; safe to read/summarize)
- `gaia mentorship --tail` — the CLI surface for direct event inspection (user-facing, identity-preserving by design)

The refusal is firm: even if the user asks "just one event", "summarize the last hour", "what fields are in there" — point to the CLI. The user owns their data; Claude does not surface it.

## Why

The mentorship stream preserves full identity for in-session adaptation (per `studio/decisions/local-adaptive-mentorship.md`). v1 ships plaintext on a `chmod 700/600` directory tree off-project; v1.1 adds OS-keychain-gated cryptographic enforcement. v1's enforcement surface is this rule — skill-instruction enforced, every GAIA skill that handles mentorship state references it.

## How to apply

Skills that handle mentorship state (gaia-init opt-in, mentorship-* CLI surface, compute-profile, adaptation injection) reference this rule by path:

> See `.claude/rules/mentorship-display.md`. Claude must not display raw mentorship event files.

That reference, plus the load via `CLAUDE.md`, makes the rule visible in every relevant context.
