# Task Orchestration

Invoked via `/gaia-plan [description]`; see [[GAIA Plan]] for the skill surface. For implementation work involving multiple files or subsystems, Claude generates a plan + orchestrator structure so each piece of work runs in a fresh-context sub-agent and the orchestrator drives the whole thing end-to-end. A spec-less one-off plan's artifacts live under `.gaia/local/plans/PLAN-NNN/` (allocated from `.gaia/local/plans/ledger.json`); a plan derived from a SPEC colocates inside the SPEC's own folder at `.gaia/local/specs/<SPEC-ID>/plan/` (`plan-2`, `plan-3`, … for a revision).

## Plan artifacts

1. **Per-task docs** in the plan folder: self-contained for fresh-context sub-agents (context, dependencies, interface contracts, files to touch, acceptance criteria). `.gaia/local/plans/PLAN-NNN/` is home for a spec-less one-off plan; a spec-derived plan colocates at `.gaia/local/specs/<SPEC-ID>/plan/`.
2. **`README.md`** with the task graph (phases + parallelism) and frozen interface contracts.
3. **`ORCHESTRATOR.md`**: full execution playbook: pre-flight branch policy, phase order with per-phase quality gates (`pnpm typecheck && pnpm lint`), sub-agent prompt template (task sub-agents pinned to `model: sonnet` by default), orchestrator-owned git flow (commits, pushes, PR), stop conditions, the phase findings ledger contract, the mandatory final summary, and a final cleanup phase.
4. **`KICKOFF.md`**: a self-contained prompt the orchestrator reads to start cold. The `/gaia-plan` skill prints the single-line resume prompt (`Read <abs-path>/<plan-dir>/KICKOFF.md and execute it.`, where `<plan-dir>` is `.gaia/local/plans/PLAN-NNN/` for a spec-less plan or the colocated `.gaia/local/specs/<SPEC-ID>/plan/` for a spec-derived one) as a fenced code block for the user to copy manually, followed by a `Type /clear and paste the prompt above.` instruction.

`PROGRESS.md` is the live run ledger (not authored at plan-write time): the orchestrator creates and appends to it as phases complete (see lifecycle step 3 below). At merge, consolidation reads `SPEC.md` → `AUDIT.md` → `PROGRESS.md` (top wins, whichever layers exist) and produces `SUMMARY.md`, the consolidated wiki-purposed artifact (see [[GAIA Plan]] for the pinned shape). A spec-less plan's `PLAN-NNN` folder is kept at merge, reduced to `SUMMARY.md` + `cost.json` with its `RUNNING` sentinel cleared; a spec-colocated plan deletes only the `plan[-N]` subfolder once its `PROGRESS.md` has fed the parent SPEC's `SUMMARY.md`, and the parent SPEC folder is kept and age-reaped later. `cost.jsonl` and the two id-ledgers are the whole durable record that survives either path.

## Execution lifecycle

When the user pastes the resume prompt, the orchestrator runs through:

1. **Resume detection.** Before pre-flight, on every cold start, the orchestrator checks the plan folder for a pre-existing RUNNING sentinel. Its absence means a fresh first run, and pre-flight proceeds unchanged. Its presence means a prior run started this same plan: after confirming the sentinel's PR isn't already merged, the orchestrator reconnects to that run's branch by the isolation mode recorded on the sentinel (checking out the branch for feature-branch isolation, re-entering the existing worktree for worktree mode), then runs a deterministic helper that verifies which phases are genuinely committed, each phase's recorded commit proven an ancestor of the reconnected HEAD. Git is the arbiter: any ambiguity re-runs the phase rather than trusting the ledger at face value. The run resumes at the first incomplete phase behind a confirmation gate (resume there, restart from Phase 1, or abandon without touching anything already done); when every phase verifies complete, resume proceeds straight to the pre-merge `code-review-audit` with no phase re-run. See [[GAIA Plan]] for the sentinel fields and the exact `PROGRESS.md` block format the helper parses.
2. **Pre-flight.** Clean working tree check. Branch policy: the orchestrator resolves isolation through the shared isolation reference, which reads the team's committed isolation policy. On `main`/`master`, that policy decides whether a prompt surfaces at all, and a team that has never set one sees the choice presented exactly as it always has. If HEAD is on any other branch, no prompt fires: worktree is the only isolation mode, the plan's work is cut into its own worktree from main rather than continuing on the current branch. Worktree creation runs through the runtime's `EnterWorktree` call; the `WorktreeCreate` hook cuts the branch fresh from main under `.claude/worktrees/` and switches the session into it, no manual `git checkout -b`. The plan folder itself is not among the gitignored state the worktree symlinks back to the main checkout, so it stays in the main checkout for the run's duration while task sub-agents edit the worktree's own copy of the tracked files they touch.
3. **Phase loop.** For each phase: dispatch sub-agents in parallel (Sonnet by default; sub-agents only edit files, they do NOT commit or push), wait for all to report, run the per-phase quality gate (`pnpm typecheck && pnpm lint`), then the orchestrator stages, commits with a meaningful message, and pushes. The PR is opened (via `gh pr create`) once the first phase's commit lands on the remote, and updated with subsequent commits; a resumed run reuses that same already-open PR rather than opening a new one. After committing, the orchestrator appends a `## Phase N, <title>` block to `PROGRESS.md` containing the phase commit SHA and the merged `Notes for orchestrator` content from every sub-agent in the phase (findings, deviations from plan, follow-ups); see [[GAIA Plan]] for the exact block format. Sub-agents emit those notes as a structured section at the end of their return; the orchestrator merges and writes; sub-agents do not write to `PROGRESS.md` directly. This append-only ledger is the durable record that survives context compression across long runs.
4. **Stop conditions.** Any sub-agent failure or quality-gate failure halts the run; the orchestrator surfaces the error to the user, appends a `## Phase N, <title> (HALTED)` block to `PROGRESS.md` with the failure context, and does NOT commit, push, or "fix and continue."
5. **Final summary.** After the last commit lands and before awaiting merge confirmation, the orchestrator reads `PROGRESS.md` and prints a brief summary: phases completed, sub-agents run, files touched (count), commits pushed (count + short SHAs), PR URL, quality-gate status, and the highest-signal findings/deviations/follow-ups drawn from the ledger so nothing is lost to compression. A few lines plus the surfaced notes, not a recap of every change.
6. **Consolidation.** Once the user confirms the PR is ready to merge, the orchestrator (or, on a cold path, `plan-close`/`spec-close`/the pre-flight backstop) produces the consolidated `SUMMARY.md`. For a spec-colocated plan, it reads `SPEC.md` → `AUDIT.md` → the colocated plan's `PROGRESS.md` (top wins) and writes the parent SPEC folder's `SUMMARY.md`; a spec-less plan's own `PROGRESS.md` produces its `SUMMARY.md` in place. Either way the result is grounded in the merged code and passing tests, present-tense, surfaces any material intent-divergence, and lands in the pinned shape (frontmatter `wiki_promote_default` + `wiki_promote_targets`, a non-empty H1, a non-empty body; see [[GAIA Plan]]). `summary-verify.sh` gates what comes next: the irreversible removal of `SPEC.md`/`AUDIT.md` proceeds only once it exits 0.
7. **Retention.** The orchestrator runs `.gaia/scripts/plan-archive.sh` on its own plan folder. A spec-less plan's `.gaia/local/plans/PLAN-NNN/` folder is kept: reduced to `SUMMARY.md` + `cost.json`, its `RUNNING` sentinel cleared, and its ledger row stamped `merged`; a folder whose consolidation has not produced `SUMMARY.md` yet is left intact instead (fail-closed). A spec-colocated plan deletes only its own `specs/<SPEC-ID>/plan[-N]/` subfolder, gated on the parent SPEC's own `SUMMARY.md` already existing so a race never destroys `PROGRESS.md` before consolidation has consumed it; the parent SPEC folder is kept and age-reaped later by the SessionStart janitor once past the retention window and cost-represented (see [[Local Working State]]). `cost.jsonl` and the two id-ledgers are the whole durable record that survives every path. `.gaia/local/` is gitignored by the GAIA default (checked via `git check-ignore`), so none of this needs a commit. If the user explicitly asks to keep the folder, the orchestrator skips the step and reports.
8. **Reconcile.** Once the orchestrator confirms the PR has merged (its own `MERGED` check, separate from the retention step's pre-merge confirmation above), and before any worktree discard, it reconciles the ledger: a spec-colocated plan flips its `specs/ledger.json` row to `merged` via `spec-reconcile.sh`; a spec-less `PLAN-NNN` plan flips its `plans/ledger.json` row to `merged` via `plan-reconcile.sh`. Both resolve the main-checkout root rather than the worktree's own copy, run best-effort, and never block. `spec-reconcile.sh` also runs as the out-of-band backstop in the `/gaia-spec` pre-flight sweep, so the ledger still converges even if this step is interrupted.

## Topology

GAIA orchestration is a depth-1 star: the main thread is the only agent that spawns, and every worker it dispatches is a leaf. Claude Code sub-agents cannot spawn further sub-agents and cannot prompt the user, so a worker never owns a team and never runs an interactive gate.

- **`/gaia-plan`** runs its thin orchestration on the invoking thread and spawns one planner leaf for the deep synthesis. The planner investigates with parallel tool calls and writes the plan files; it does not spawn sub-agents.
- **The execution orchestrator** (a fresh session started from `KICKOFF.md`) is itself a main thread, so it dispatches the per-phase implementation sub-agents as leaves and runs the pre-merge `code-review-audit`.
- **`/gaia-spec`** ends by printing a `/gaia-plan` handoff prompt and stops; it spawns nothing downstream. The human runs `/gaia-plan` in a fresh session, where the planner is the single spawned leaf.

Models pin at spawn, so the main thread, even on Sonnet, puts the synthesis on a top-tier model (Opus or Fable) by spawning the planner with an explicit `model` override. The execution orchestrator pins the opposite direction: each per-phase implementation sub-agent spawns with `model: sonnet`, because the spec and plan audits resolve the feature's complexity upstream so execution runs on the cheaper model regardless of the orchestrator's own session model. The planner escalates a specific phase to Opus only when it names a genuinely deep-synthesis reason. Interactive steps stay on the main thread because only it can prompt.

## Why

- Keeps each sub-agent's context focused.
- Avoids massive multi-file edits in one pass.
- User can review the plan before committing compute.
- Individual tasks are resumable / re-runnable.
- The orchestrator owning git means commit history reflects phase boundaries cleanly, and broken state never reaches the remote.
